# ISUBID 設計ドキュメント

作成日: 2026-07-08
ステータス: 承認済み

## 概要

**ISUBID** は、ISUCON形式の性能チューニング競技問題(お題Webアプリ + ベンチマーカー)の自作プロジェクト。題材は「椅子専門のライブオークションサイト」。OSSとしてGitHubに公開する。

### 出題コンセプト

- **王道のDBチューニング**(インデックス、N+1、集計クエリ、BLOB外出し)と、**pub/sub的なアーキテクチャ要素**(入札のリアルタイム配信、通知ファンアウト、非同期終了処理)の両方を出題する
- すべての処理が本物のデータ操作として成立する(模擬・ダミー処理なし)。「入札の整合性を壊すとFailする」制約下でどこまで攻めるかが醍醐味
- 過去のISUCON出題(チャット=7、チケット予約=8、フリマ=9、不動産検索=10、IoT監視=11、ゲームリーダーボード=12、ライブ配信=13、ライドシェア=14)と被らない題材

### 主要な決定事項

| 項目 | 決定 |
|---|---|
| 題材 | ライブオークション(椅子専門) |
| アプリ名 / リポジトリ名 | ISUBID / `isubid` |
| 参照実装 | Go 先行。多言語(Rust/Ruby/Python等)を後から追加できる構成 |
| ベンチマーカー | Go + [isucandar](https://github.com/isucon/isucandar)(ISUCON10〜13公式ベンチの土台。MIT) |
| UI | あり。言語実装間で共有するSPA(ビルド済みを配置) |
| DB | MySQL 8(本家準拠。ISUCON14も `mysql:8`) |
| ミドルウェア | nginx + app + MySQL |
| 実行環境 | Docker Compose 先行、クラウドIaC(複数台構成)は後日 |
| 用途 | OSS公開(ドキュメント・再現性・レギュレーションまで整備) |
| 構築方針 | 段階構築(Phase 1〜5)。各Phase末に参照実装へベンチをかけて検証 |

## リポジトリ構成

```
isubid/
├── README.md              # コンセプト・クイックスタート
├── docs/
│   ├── regulation.md      # レギュレーション(禁止事項・スコア計算式)
│   ├── manual.md          # 当日マニュアル(アプリ仕様の説明)
│   └── writeup.md         # 仕込んだボトルネック解説(ネタバレ、別置き)
├── webapp/
│   ├── frontend/          # 共有SPA(ソース。ビルド成果物を public/ に配置)
│   ├── go/                # Go参照実装(初期実装=わざと遅い)
│   ├── public/            # ビルド済みフロントエンド+静的アセット
│   └── sql/               # スキーマ+初期データ投入
│       ├── init.sh        # ベンチ前初期化(POST /initialize から呼ばれる)
│       └── *.sql
├── bench/                 # ベンチマーカー(Go + isucandar)
├── initial-data/          # 初期データジェネレータ(Go)と生成物
├── dev/                   # docker compose 一式(MySQL/nginx/app)
└── provisioning/          # クラウドIaC(後日フェーズ)
```

- `webapp/go` を言語別に増やせる構造(将来 `webapp/rust`, `webapp/ruby`, `webapp/python`)。スキーマ・SQL・フロントエンド・nginx設定は言語間で共有
- フロントエンドはSPA。ビルド済みを `public/` にコミットし、どの言語実装でも同じUIが動く(ISUCON13方式)
- 初期データはジェネレータで生成し、成果物(SQLダンプ)をコミット(大きければGitHub Releases)。乱数シード固定で再現可能
- `docs/writeup.md` に仕込んだボトルネックの解説を残す(OSSとしての価値+答え合わせ用)

## お題アプリ「ISUBID」仕様

**コンセプト**: 椅子専門のライブオークションサイト。出品された椅子に参加者がリアルタイムで入札を競り合う。

### エンティティ

- `users` — 出品者/入札者(アイコン画像あり)
- `auctions` — 出品(タイトル、説明、カテゴリ、開始価格、開始/終了時刻、状態: `upcoming/live/closed`)
- `bids` — 入札(auction_id, user_id, 金額, 時刻)。最重量テーブル
- `notifications` — 通知(高値更新された、落札した、など)
- `categories` — カテゴリマスタ

### API(+SPA画面)と仕込むボトルネック

| 機能 | エンドポイント | 仕込むボトルネック |
|---|---|---|
| 初期化 | `POST /initialize` | (ベンチ用) |
| 登録/ログイン | `POST /register`, `POST /login` | bcryptコスト過剰 |
| トップ: 開催中一覧 | `GET /auctions` | 現在価格をN+1で取得、入札数をCOUNTサブクエリ |
| 検索/カテゴリ絞り込み | `GET /auctions?category=&q=` | `LIKE '%q%'` フルスキャン、index欠如 |
| オークション詳細 | `GET /auctions/:id` | 入札履歴全件取得+ユーザーJOINのN+1 |
| **入札** | `POST /auctions/:id/bids` | 現在最高額チェックを `SELECT MAX()` 全件走査+ロックなし(正しさはギリギリ担保される遅い実装) |
| **入札フィード** | `GET /auctions/:id/bids?since=` | 1秒間隔ポーリング前提の重いクエリ。**pub/sub化ポイント**(SSE/Long-poll/インメモリ配信へ進化させる) |
| 通知一覧 | `GET /notifications` | 高値更新のたび全入札者にINSERTするファンアウトが入札APIを重くしている |
| 終了処理 | (アプリ内バッチ) | 毎秒ポーリングで終了auction検索→落札確定処理が同期実行で重い |
| ユーザーアイコン | `GET /users/:id/icon` | 画像をDBのBLOBに保存(静的配信化ポイント) |
| 売上ダッシュボード | `GET /stats/me` | 出品者の売上集計を毎回全走査 |

### 改善の道筋(出題意図)

1. 序盤: インデックス追加・N+1解消・画像BLOB外出しの王道
2. 中盤: 通知ファンアウトの非同期化
3. 終盤: 入札フィードのpub/sub化、入札処理の競合最適化

入札の整合性(高値更新の単調増加・落札者の正しさ)を壊すとベンチがFailする制約が全体を貫く。

## ベンチマーカー設計

isucandarベースのGo製。実行形式は `bench -target http://localhost:8080`。負荷走行は60秒(本家標準)。

### isucandarの利用パッケージ

- `agent` — 仮想ユーザー1人=1 agent。Cookie管理、条件付きGET(304)、`ProcessHTML()` によるアセット追従取得(UI検証)
- `worker` — シナリオの並列実行、負荷の段階増加
- `score` — タグ別スコア集計
- `failure` — エラー分類(致命的/減点のみ)
- `parallel` — 検証ジョブの並行実行
- `pubsub` — シナリオ間連携(新規出品を入札者シナリオへ通知)

### 3フェーズ

1. **Prepare** — `POST /initialize`(タイムアウト30秒)。その後、少数リクエストで全エンドポイントの応答形式・初期データの正しさを検証。違反したら即Fail(負荷走行に進まない)
2. **Load**(60秒) — シナリオ群をworkerで並列実行。エラー率とスコアに応じて並列数を段階的に増やす(速いアプリほど強い負荷が来る本家方式)
3. **Validation** — 走行終了後、`closed` になったオークションの落札結果を照合。ベンチは送信した全入札を記録しており、「最高額入札者が落札者になっているか」「落札価格が正しいか」を突き合わせる

### シナリオ

- **入札者シナリオ**(主役): ログイン → トップ閲覧 → オークション選択 → 詳細+入札フィードをポーリングしつつ、競り負けたら上乗せ入札。終了間際の人気オークションに人が集まる挙動を再現
- **ウォッチャーシナリオ**: 入札せず一覧・検索・詳細を回遊(read負荷の主源。アセット取得込みでUI検証)
- **出品者シナリオ**: 新規出品 → 売上ダッシュボード確認。新規出品は `pubsub` で入札者シナリオへ通知され、入札が集まる
- **通知確認シナリオ**: 高値更新された入札者が通知一覧を見に来る

### 整合性検証(Failの条件)

- 入札成功レスポンスを返したのに履歴に現れない/金額が現在価格以下で成功した(単調増加違反)
- 落札者・落札額がベンチの記録と不一致
- 自分の入札がN秒(初期値: 2秒)以内にフィードへ反映されない(リアルタイム性要件。ポーリング間引きによるズルの防止)
- 通知の欠落(高値更新されたのに通知が来ない)
- 致命エラー(5xx・検証違反)は種類ごとに閾値超えでFail。タイムアウトは減点扱い

### スコア計算

```
スコア = Σ(成功リクエスト × タグ別配点) − 減点(エラー・タイムアウト)
配点例: 入札成功=5, 出品=5, 詳細閲覧=1, 一覧=1, 通知確認=2
```

入札が高配点であり、「入札処理と入札フィードを速くする=スコアが伸びる」と出題意図が一致する。結果はJSON+人間可読サマリで出力。

## 初期データ

`initial-data/` のGoジェネレータ(シード固定)で生成し、SQLダンプを成果物化。

- ユーザー約5,000人(アイコンPNGはジェネレータで生成、DBのBLOBに格納)
- 終了済みオークション約10,000件+入札約30万件(過去データの重さで一覧・集計を遅くする)
- 開催中(live)約200件、開始待ち(upcoming)約100件。ベンチ走行中に順次 live→closed へ遷移するよう終了時刻を分散配置
- 「初期データの正解スナップショット」(各オークションの現在価格など)をJSONで持ち、Prepareフェーズの照合に使う

## 実行環境

### Docker Compose(先行)

`dev/compose.yaml` で一発起動:

- `nginx`(:8080) — 静的ファイル配信もあえてアプリ経由にした遅い初期設定
- `app` — `webapp/go` をビルドして起動(環境変数で言語実装を差し替え可能)
- `mysql`(8.x) — `innodb_buffer_pool_size` 等デフォルトのまま(チューニング余地)

ベンチはホストから実行(コンテナ版も用意)。compose内のアプリ・ミドルウェア設定は自由に改造可のレギュレーション。`deploy.resources` でCPU/メモリ制限をかけられるようにしてマシン差を吸収。

### クラウドIaC(後日)

初回リリース後に `provisioning/` へ複数台構成のIaCを追加する。

## 開発の進め方

### フェーズ分け

1. **Phase 1 — 骨格疎通**: リポジトリ初期化、スキーマ+最小Goアプリ(`/initialize`・認証・一覧・詳細・入札)、compose環境、ベンチはPrepareフェーズのみ(整合性チェッカーとして動く状態)
2. **Phase 2 — 負荷走行**: 入札者/ウォッチャーシナリオ+スコアリング+Load 60秒走行。初期データジェネレータ(縮小版)。ここで初めてベンチが回りスコアが出る
3. **Phase 3 — pub/sub要素**: 入札フィード(反映N秒検証)、通知ファンアウト、終了処理と落札Validation、出品者シナリオ
4. **Phase 4 — 仕上げ**: SPAフロントエンド+アセット検証、アイコンBLOB、検索、初期データフルサイズ、負荷の段階増加、レギュレーション文書
5. **Phase 5(リリース後)**: 多言語参照実装、クラウドIaC

各Phaseの終わりに「参照実装にベンチをかけてFailしない+スコアが再現する」ことを確認してから次へ進む。

### テスト戦略

- ベンチの検証ロジック(入札整合性チェッカー、スコア計算)はユニットテストを厚めに書く(ベンチ自体の正しさが最重要)
- **ベンチのベンチ**: わざと壊した実装(入札の単調増加チェックを外す、フィード反映を5秒遅らせる等)に対してベンチが正しくFailを出すことを確認するテストを用意。OSSとしての信頼性の核
- アプリ側はハンドラのHTTPテスト最小限+ベンチPrepareフェーズが実質的な結合テストを兼ねる
- CIは縮小データ+短時間走行(10秒)のスモークベンチ

## 検討して不採用にした選択肢

- **題材: オンラインジャッジ/コンテスト** — ジャッジの実体が模擬(ダミー計算)になり、チューニングの手応えが空虚になるため不採用
- **題材: ライブ配信コメントサイト** — ISUCON13(ISUPipe)と被るため不採用
- **ベンチ完全自作** — worker管理・ブラウザ模擬の再発明に工数を取られるため不採用。isucandar採用でシナリオ設計・整合性検証・スコアリングに集中する
- **PostgreSQL** — 本家準拠(MySQL 8)を優先。歴代ISUCONにPostgreSQL採用例はない
